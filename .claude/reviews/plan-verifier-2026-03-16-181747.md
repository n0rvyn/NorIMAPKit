### [Assertion 1] Task 1: Package.swift declares wrong package for MLXLLM/MLXLMCommon products

The plan's Package.swift (line 62-97) declares dependency on `mlx-swift` only:
```swift

```
Then tries to use products from it:

**Verification:** Read mAil's `Package.resolved` -- `MLXLLM` and `MLXLMCommon` come from `mlx-swift-examples` (identity: `mlx-swift-examples`, location: `https://github.com/ml-explore/mlx-swift-examples`). **Result:** :x: Assertion stands. The Package.swift will fail to resolve `MLXLLM` and `MLXLMCommon` products from `mlx-swift`.

**Required revision:** Task 1 must add `mlx-swift-examples` (or `mlx-swift-lm`) as a dependency, and map `MLXLLM`/`MLXLMCommon` products to that package. The dev guide itself mentions `mlx-swift-lm` as a dependency (line 21: "mlx-swift, mlx-swift-lm, swift-transformers"). The plan contradicts the dev guide.

### [Assertion 3] Task 10: Test `testCloudBackendConfigMindOrbit` has invalid Swift syntax for enum pattern matching

Plan line 1386:
```swift
if case .appAttest = config.authMethod else {
    XCTFail("Expected appAttest auth method")
}
```

This is not valid Swift syntax. `if case` requires a body for the matching case; the `else` block runs when it does NOT match, but the `if` branch needs at least `{}`. The correct form would be:
```swift
guard case .appAttest = config.authMethod else {
    XCTFail("Expected appAttest auth method")
    return
}
```
Or alternatively using `if case ... { } else { XCTFail(...) }`.

**Result:** :x: Assertion stands. Test will not compile.

**Required revision:** Task 10, `testCloudBackendConfigMindOrbit` -- fix the `if case` syntax to use `guard case` with `return`, or use proper `if case ... { } else { ... }` form.

**Verification:** Read `mAil/Services/LocalLLMBackend.swift:150` -- confirmed `GPU.clearCache()` is called after generation completes. This prevents Metal memory accumulation.

**Result:** :x: Assertion stands. Functional regression: Metal GPU cache will grow unbounded across generation calls.

**Required revision:** Task 6, `doGenerate` method -- add `GPU.clearCache()` call after `withTaskGroup` returns, matching mAil's implementation.

### [Assertion 5] Task 3: GenericClassificationRequest/Response do not conform to ClassificationRequestType/ClassificationResponseType protocols

Plan defines `ClassificationRequestType` protocol (plan line 194-201) and `ClassificationResponseType` protocol (plan line 204-210), then creates "default implementations" `GenericClassificationRequest` (line 216-230) and `GenericClassificationResponse` (line 232-243). Neither struct declares conformance to their respective protocol.

**Verification:** `GenericClassificationRequest` has matching properties (`itemID`, `title`, `body`, `timestamp`) but no `: ClassificationRequestType` declaration and no `Category` associated type. `GenericClassificationResponse` has `category: String` but `String` does not conform to `ClassificationCategory` (which requires `RawRepresentable where RawValue == String`).

**Result:** :x: Assertion stands. The "default implementations" are standalone structs disconnected from the protocol hierarchy. They cannot be used in generic code expecting `ClassificationRequestType` or `ClassificationResponseType` conformance.

**Required revision:** Task 3 needs to either: (a) make the generic types conform to the protocols (requiring a `GenericCategory` enum conforming to `ClassificationCategory`), or (b) explicitly document these as non-protocol convenience types for simple string-based use cases. Option (a) is more consistent with the DP-005 decision. Note: `String` cannot directly conform to `ClassificationCategory` because `String.RawValue` is not `String` (String is not `RawRepresentable`).

### [Compile Failure 3] Task 6 non-arm64 stub missing `init` parameters

**Assumed failure:** The non-arm64 stub (plan line 571-578) declares `public init() {}` with no parameters. The arm64 version (line 439) has `init(modelDirectory: URL? = nil, modelID: String = "qwen3-1.7b")`. Callers that pass arguments to the init (e.g., from LLMService) will fail on non-arm64 because the stub init accepts no parameters.

**Plan coverage:** Partial. The LLMService (Task 8) uses factory closures, so the init mismatch may not surface if factories are arch-guarded. However, if any caller constructs `LocalLLMBackend(modelDirectory: ...)` on non-arm64, it fails. This is a latent risk.

### [Runtime Regression 1] ModelDownloadService uses @unchecked Sendable with mutable state

**Assumed regression:** `ModelDownloadService` is marked `@unchecked Sendable` (plan line 1003) with multiple mutable properties (`phase`, `overallProgress`, `fileProgress`, `bytesReceived`, etc.) that are mutated from both `@MainActor` delegate callbacks and the `start()`/`cancel()` methods. The original mAil version is `@MainActor @Observable`, which serializes all access on MainActor.

The plan version removes both `@MainActor` and `@Observable`, replacing with `@unchecked Sendable`. The URLSession delegate callbacks dispatch to `@MainActor` via `Task { @MainActor in ... }`, but `self.updateProgress(...)` and `self.handleDownloadCompletion(...)` are called on `@MainActor` while the class itself is not `@MainActor`. If `start()` or `cancel()` is called from a non-MainActor context, there is a data race.

**Plan coverage:** :x: Not covered. The plan should either keep `@MainActor` (matching mAil) or make it an actor.
